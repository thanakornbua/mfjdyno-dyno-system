# Security / Bug / Leak Audit — July 2026

Scope: `crates/dyno-core` (Rust backend), `java/` (JavaFX operator console), deploy assets.
Method: read-then-verify; only confirmed issues listed. Fixes are minimal patches per project convention.

## Fixed in this audit

| # | Severity | Area | Finding | Fix |
|---|----------|------|---------|-----|
| 1 | High | storage.rs | Configured license plate/customer never reached the `runs` row — run-control state and the storage recorder were disconnected, so history was unsearchable by plate. | Pending-run-metadata command stamps plate/customer onto the run row at creation (Phase 1 commit). |
| 2 | Medium | storage.rs | `close_run` stamped wall-clock time instead of the closing frame timestamp, corrupting run duration for replayed/buffered frames and failing the existing lifecycle test. | Close with `frame.ts_ms`. |

| 4 | Medium | api.rs | No length limits on operator-supplied strings (plate, customer, notes); oversized payloads were stored verbatim. | `validate_text_field` caps plate 32, names 128, phone 32, notes/search 1024/128 chars on configure and PATCH. |
| 5 | Low | LiveDynoChartPresenter | Live chart series grow unbounded for the duration of a recording; an abnormally long (or never-stopped) recording grows heap at 20 Hz × series. | Capped at 20,000 points per series (~17 min at 20 Hz). |

## Verified sound (no change needed)

- **SQL injection**: every user-facing query in `storage.rs` uses bound parameters; the only `format!`-built SQL is internal migration DDL with constant identifiers. The new search endpoint uses `instr(upper(...), upper(?1))` with a bound parameter.
- **Dev API**: `/api/dev/seed-run` is rejected unless `DYNO_ENABLE_DEV_API=true` or a debug build.
- **WebSocket backpressure**: broadcast uses a `watch` channel (latest-value only) — slow clients skip frames instead of queueing; per-client tasks exit on send failure, so dead connections don't leak.
- **Serial ingestion**: fresh line buffer per read, read timeouts, bounded consecutive-failure counter before the port is closed and reopened; malformed JSON lines are skipped, not panicked on.
- **Storage worker**: single blocking thread with a bounded (1024) command queue; active run is closed on shutdown.
- **Java HTTP clients**: 2 s connect / 5 s request timeouts; API calls run off the FX thread with results marshaled back via `Platform.runLater`.
- **WS client (Java)**: single daemon scheduled-executor thread with capped-backoff reconnect — no thread leak, doesn't block JVM exit.
- **CSV/JSON exports**: try-with-resources writers.
- **Export filenames**: derived from numeric run id and user-picked via `FileChooser`; no path-injection surface.
- **Systemd units / env files**: no secrets stored; `Restart` policies present.

## Known residual risks (accepted, documented)

- No authentication on the HTTP/WS API — mitigated by loopback-bind guidance; adding auth is a feature, not a patch.
- System password stored in plaintext in SQLite `settings` — acceptable for a shop-floor appliance behind the same trust boundary as the DB file itself.
- `frames` retention is manual (delete runs + VACUUM); no automatic pruning.
